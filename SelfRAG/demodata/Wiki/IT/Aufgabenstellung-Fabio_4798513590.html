Created by Reto Bättig, last modified by nizami.sevindi on Aug. 26, 2024

1.    Einleitung

Cudos benutzt für die Softwareentwicklung verschiedene VCS (Version Control System)

  • Subversion (SVN): Interner Server (http://intrasvn.mfen-dom.local/svn_repos.php)
  • Team Foundation Server (TFS)
    • Interner Server (http://mfwinsrv03:8080/tfs/MfMainCollection)
    • Cloudbasiert: Visual Studio Team Serivces: https://www.visualstudio.com/de/team-services/ für das Ionplus Projekt (https://ionplus.visualstudio.com/)
  • GIT
    • Cloudbasiert: Github
    • Cloudbasiert: Bitbucket

 Viele Mitarbeiter / Projektleiter würden Projekte lieber mit GIT machen als mit SVN oder TFS, wollen oder können (aus Sicherheitsgründen) aber nicht auf einen gehosteten Service gehen.

2.    Ziele

  1. Abklären des Bedarfs und der Anforderungen an einen internen GIT-Server
    1. Folgende Personen befragen: EWC, RAM, MEM und Anforderungen/Wünsche sammeln
    2. Anforderungen in Wiki-Seite dokumentieren
  2. Abklären der verschiedenen Möglichkeiten, intern einen GIT-Server aufzusetzen
    1. Anforderungen an Hardware / Betriebssystem
    2. Kosten (Fix, Wiederkehrend)
    3. Initialaufwand für Installation, Konfiguration und Dokumentation
    4. Weitere Limitierungen, Bedingungen?
    5. Benötigte Ressourcen, Zeitplan
  3. Entscheid über Umsetzung / Varianten (BAR, ev. im Team)
  4. Falls GO: Umsetzung
    1. Aufsetzen des Servers
    2. Dokumentation
    3. Test
    4. Übergabe an IT (ZEH, BAR, BRS)
    5. Schulung Mitarbeiter

3.    Anforderungen/Wünsche aus Sicht BAR

  • Wenn möglich auf SVN-Server laufen lassen (MFWSRV02, Windows 7 Professional)
  • Automatisches Backup muss integriert sein in in unsere Backup-Strategie (mit Hans Zemp absprechen)
  • Erstellung von neuen Repositories, Zugriff etc. muss dokumentiert sein und einfach von Mitarbeitern selbst durchführbar sein
  • Authentisierung falls möglich über Domänen-Login (nice to have)
  • Prozess, um ein Repository von Bitbucket nach Cudos Intern zu transferieren (und ev. umgekehrt) soll definiert, getestet und beschrieben sein
  • Eine Einführung in GIT soll für alle Mitarbeiter im Rahmen eines Tech-Talks nach der MA-Info (1h) gemacht werden. Grundkonzepte und Verwendung sollen gezeigt und demonstriert werden.
